## 🔴 NEW SINCE LAST AUDIT

### Reduced Motion — DatePicker regression

**DatePicker.tsx** uses `motion` from `motion/react` (scale animations on buttons at lines 220, 313, 333, 405; x/opacity slide transition at line 535) but does **not** import or use `useReducedMotion`. All other motion-using components (Skeleton, Checkbox, Table, Modal, DropdownMenu, Tabs, Toggle, RadioButton, Button) correctly use the hook.

| File | Lines | Code | Fix |
|---|---|---|---|
| **DatePicker.tsx** | 220, 313, 333, 405 | `animate={{ scale: isPressed ? 0.9 : 1 }}` | Wrap scale in `reducedMotion ? 1 : ...` |
| **DatePicker.tsx** | 535 | `animate={{ opacity: 1, x: 0 }}` | Under reduced motion: use opacity-only fade, no `x` translate |

**Fix:** Import `useReducedMotion` from `../hooks/useReducedMotion`, call it in the component, and gate all spatial animations (scale, x) behind the reduced motion check.

### 2. Hardcoded Shadows (2 violations)

| File | Line | Code | Fix |
|---|---|---|---|
| **Button.tsx** | 166 | `const SHADOW_MEDIUM_1 = '0px 10px 20px 0px rgba(0,0,0,0.08)'` | Import and use `shadows['medium-1']` from `tokens/shadows` |
| **DropdownMenu.tsx** | 46 | `const SHADOW_LARGE = '0 20px 40px rgba(0,0,0,0.32)'` | Import and use `shadows['large-3']` from `tokens/shadows` |

Both components also define `SHADOW_HIDDEN = 'none'` — this is acceptable (not a token violation).

### 3. Border Radius — `rounded-full` (7 occurrences)

| File | Lines | Context |
|---|---|---|
| **RadioButton.tsx** | 201, 222, 235 | Radio circle outer, inner dot, ghost dot |
| **Toggle.tsx** | 153, 168 | Track and thumb |
| **NumberInput.tsx** | 193, 257 | Increment/decrement buttons |

RadioButton and Toggle use `rounded-full` for inherently circular elements (radio dots, toggle thumbs/tracks). These are **architecturally justified** — a radio button and toggle thumb must be round. Consider documenting this as an accepted exception in `docs/conventions.md`.

NumberInput's stepper buttons using `rounded-full` is less justified — **replace with `rounded-m`** for consistency.

### 4. Motion — 1 Override

| File | Line | Code | Fix |
|---|---|---|---|
| **Skeleton.tsx** | 138 | `bounce: 0.15` (overrides `spring.playful`'s 0.25) | Create a dedicated `spring.subtle` token or remove the override |

### 5. Semantic HTML (minor, carried forward)

| File | Line | Code | Fix |
|---|---|---|---|
| **NumberInput.tsx** | 213 | `type="text"` on number input | Change to `type="number"` or `inputMode="numeric"` for mobile keyboards |

### 6. Focus Ring Pattern Note

Input, TextArea, NumberInput, and SearchInput use a JS-driven `focused` state to change border styling (via `boxShadow`) rather than the `focus-visible:outline` pattern specified in conventions. These DO provide a visible focus indicator on focus, but use a different mechanism. This is a common pattern for text fields (where you want visible focus even on click, not just keyboard), so it is **not counted as a violation** — but worth noting for convention consistency.
